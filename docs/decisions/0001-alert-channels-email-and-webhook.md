# 0001. 알림 채널: 이메일 유지 + Slack/Discord webhook 추가

- 상태: 채택 (2026-07-08, 같은 날 두 차례 수정)
- 결정자: 신현수

## 배경

초기 계획(AGENTS.md)은 "이메일 알림으로 시작하고 Slack은 나중에 추가"였다. 이에 따라 이메일 발송 파이프라인(pending alert 모델, SMTP 발송 worker task, FAILED 재시도 API, 웹 수신자 설정 UI)이 구현됐지만, 실제 발송은 운영 환경 SMTP 자격증명 설정이 필요해 비활성 상태였다.

운영 준비 과정에서 **이메일을 제거하고 webhook 단일 채널로 교체하는 안**을 먼저 검토했다. 근거는 다음과 같았다.

- AIM의 타깃 사용자(1인 개발자·소규모 팀)의 실시간 장애 인지 채널은 이메일이 아니라 Slack/Discord다.
- SMTP 발송은 발신 도메인 평판, 스팸 필터, 자격증명 관리 등 1인 운영 규모 대비 부담이 크다.
- incoming webhook은 무료이고 설정이 URL 등록 한 줄로 끝난다.

그러나 같은 날 검토 과정에서 **비밀번호 재설정 등 트랜잭션 메일에는 이메일 발송 수단이 어차피 필요**하다는 점이 확인됐다. SMTP 인프라를 유지해야 한다면 "이메일 제거"의 핵심 이점(SMTP 운영 부담 해소)이 사라지고, 이미 완성된 이메일 알림 파이프라인을 제거하는 비용만 남는다.

## 결정

**이메일 알림 파이프라인은 유지하되 검사 결과(incident) 이메일은 기본 off로 두고, Slack/Discord incoming webhook을 알림의 1차 채널로 도입한다. 이메일의 1차 용도는 인증/트랜잭션 메일이다.**

- 이메일 발송 파이프라인과 SMTP 설정: 유지. 1차 용도는 향후 구현할 인증 메일(가입 확인·비밀번호 재설정 등)이다.
- 검사 결과 알림 이메일: `alert_email_enabled` 기본값을 False로 변경(모델·스키마·DB server_default). 프로젝트별 opt-in으로만 발송한다. 운영 DB의 기존 프로젝트도 2026-07-08에 모두 off 처리했다.
- webhook 채널(도입 예정): 프로젝트별로 Slack/Discord incoming webhook URL을 등록하고, 등록된 URL로 alert payload를 전송한다.
- webhook URL도 사용자 입력 URL이므로 기존 SSRF-safe 검증을 동일하게 적용한다.
- alert trigger 규칙, incident open/recovery 파이프라인, pending alert 저장 구조는 채널과 무관하게 공유한다.

## 검토한 대안

1. **webhook 단일 채널 (이메일 제거)** — SMTP 운영 부담 해소가 목적이었으나, 트랜잭션 메일 때문에 SMTP가 어차피 필요해 이점이 사라진다. 완성된 이메일 파이프라인 제거 비용과 마이그레이션만 남는다. 기각.
2. **이메일 단일 유지 (기존 계획)** — 타깃 사용자의 실시간 인지 채널로 부족하다. 기각.
3. **이메일 알림 기본 on 유지 + webhook 추가** — 임계값 위반 시 사용자가 예상하지 못한 메일이 발송되는 경험이 좋지 않고, 알림의 1차 채널은 webhook으로 갈 예정이므로 이메일 알림을 기본으로 켤 이유가 없다. 기각.
4. **이메일 파이프라인 유지(기본 off, opt-in) + webhook 1차 채널 + 이메일은 인증 메일 중심** — 채택.

## 결과와 영향

- 검사 완료·incident 발생 시 이메일은 프로젝트에서 명시적으로 켠 경우에만 발송된다.
- SMTP 설정은 유지하며, 향후 비밀번호 재설정 등 인증 메일이 같은 인프라를 사용한다.
- webhook 채널 구현 범위(별도 작업):
  1. 프로젝트별 webhook URL 등록 API·설정 UI (SSRF-safe 검증 포함)
  2. webhook 전달 worker task (실패 시 재시도 — 기존 pending alert·재시도 구조 재사용)
  3. README 등 구현 상태 문서 갱신
- 이메일 경로 제거·마이그레이션은 하지 않는다.
